---
name: Supabase-only CRM architecture
description: Why the CRM must never call the Express API server and how to do mutations correctly.
---

## Rule
All CRM data hooks (useCustomers, useBills, useProducts, useInventory, etc.) must use the Supabase JS client directly. Never import from `@/lib/api` or call `/api/...` routes.

**Why:** The API server (`artifacts/api-server`) uses Replit OIDC (passport-based session auth). After switching to Supabase email/password auth, the API server will return 401 for every protected route. The Vite proxy for `/api` was also removed.

## How to apply
- For all `.from()` calls that do INSERT/UPDATE/DELETE, cast as `(supabase as any).from(...)` — the typed client can't infer mutation shapes for partial types.
- For SELECT queries, you can also use `(supabase as any)` to avoid TypeScript narrowing issues with `.select("col1, col2")` partial selects.
- Bill creation: use multi-step Supabase calls (insert bill header → insert bill_items → call `record_stock_movement` RPC per item → insert payment → update customer denorm fields). The `create_invoice` RPC exists but its payload format is unknown; manual steps are safer.
- Bill number is auto-generated by a DB trigger — do not set it on insert.
- The `payments.payment_method` enum does NOT include "credit" (bills does). Normalize "credit" → "cash" when writing to payments.

## Key tables
- `customer_notes`, `customer_photos`, `house_mappings`, `customer_paint_shades` — all have `customer_id` FK
- `payments.payment_date` — date string field (not `created_at`)
- `bills.status` enum includes "unpaid", "partially_paid", "paid", "overdue", "cancelled", "sent", "draft"
